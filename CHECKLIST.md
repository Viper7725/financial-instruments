# 项目检查清单

这份清单用于约束本项目后续迭代，避免重复犯错。

## 我容易犯的错误

1. 过早下结论
- 在原始明细没有逐笔闭环前，不提前判断“结果没问题”或“规则已经修好”。

2. 把样本跑通当成规则普适
- 必须区分“当前样本已修复”和“已经形成通用规则”。
- 如果只是修复了某一份样本，要明确标注适用范围。

3. 对命名、月份识别、输出路径验证不够
- 任何涉及文件名、月份分类、默认输出路径的修改，都必须拿真实样本文件名实际验证一次。

4. 对异常数据形态预判不足
- 不能只相信标准列。
- 要优先考虑“订单号在备注里”“业务描述为空”“列值缺失但别列可补足”的情况。

## 以后必须执行的检查

1. 明细闭环检查
- 结论类字段有争议时，必须回到原始账单逐笔核对。
- 至少要能说明“哪一行进了哪个分类，为什么”。

2. 样本验证检查
- 每次修规则后，至少用当前用户样本重跑一次。
- 如果修改的是通用逻辑，要额外抽一份历史样本做回归验证。

3. 输出检查
- 必须确认输出文件实际落在正确月份文件夹。
- 必须确认输出文件名符合“对账结果-YYYY年MM月.xlsx”。

4. 异常归类检查
- 对无法直接归类的金额，要确认它进入了“其他收入 / 其他支出 / 其他手续费 / 其他基础软件费”中的哪一类。
- 不能只说“进了其他”，要说清楚进的是哪一列。

5. 表达检查
- 对用户的说明里，要明确区分：
- 已确认
- 推测
- 待复核

## 当前项目特别注意

1. 手续费不能只靠业务描述列识别。
- 如果备注里带有订单号和“天猫佣金（类目系统）扣款”信息，也要尝试按订单回填游戏。

2. 退款缺少负分成冲回时，推理结果只能放在参考 sheet 里，不能偷偷覆盖原始汇总。

3. 如果输出结果和财务口径不一致，优先先查原始明细，不先和财务争论口径。
